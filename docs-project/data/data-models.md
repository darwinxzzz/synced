# Data Models

## Overview
This document describes the core data entities, their relationships, and validation rules used in Synced.

Synced stores application data in Supabase Postgres. The generated database contract lives in `src/types/database.ts` and exports:

- `Database` — full Supabase schema type
- `Tables<T>` — row type for a table
- `TablesInsert<T>` — insert payload type for a table
- `TablesUpdate<T>` — update payload type for a table

Generated types currently list no Postgres enum types and no explicit relationship metadata. Relationships below are inferred from column names such as `user_id`, `event_id`, and `contribution_id` and should be refined against Supabase migrations or database constraints when needed.

### Kanban
The generated Supabase types do not currently include a dedicated `kanban` table. Kanban-style task data appears to be represented by event/member task fields and contribution records:

- `event_members.task` for event-specific assignments
- `event_members.pillar_status` for milestone/status tracking
- `contributions.task`, `contributions.priority`, `contributions.outcome`, and `contributions.submitted_at` for contribution/task lifecycle data

Future refinement should document any application-level Kanban interfaces or persisted board structures if they are added outside the generated database schema.

## Relationships
Generated type metadata currently reports `Relationships: []` for every table, so the following map is inferred from foreign-key-like column names:

```text
profiles.id
  ├─ events.created_by
  ├─ event_members.user_id ─────┐
  ├─ attendance.user_id         │
  ├─ contributions.user_id      │
  │    └─ reflections.contribution_id
  ├─ reflections.user_id
  ├─ testimonial_requests.user_id
  └─ testimonials.user_id

events.id
  ├─ event_members.event_id
  ├─ attendance.event_id
  └─ contributions.event_id
```

Common inferred cardinalities:

- One profile can create many events.
- One event can have many event member rows.
- One profile can have many event member rows, attendance rows, contributions, reflections, testimonial requests, and testimonials.
- One event can have many attendance rows and contributions.
- One contribution can have many reflections, depending on database constraints not visible in generated types.

## Validation
- Zod schemas are used for service-layer input validation before database writes.
- Example: testimonial validation schemas in `src/server/services/testimonials/schemas.ts` validate testimonial request/submission payloads.
- Database types provide compile-time shape checks through `Tables`, `TablesInsert`, and `TablesUpdate` but do not replace runtime validation.
- RLS policies enforce data access control at the database layer, with application-level authorization using auth types such as `UserRole`, `AccessProfile`, and `AuthState` in `src/lib/auth/access.ts`.

## Data Flows
- **Event creation:** authorized user submits event details and member ids → tRPC/server service validates and calls Supabase, optionally via the `create_event` SQL function → `events` row is created → related `event_members` rows associate members with the event.
- **Attendance tracking:** user/event/date attendance data is submitted → service validates allowed status/type values where applicable → `attendance` row is inserted or updated under RLS.
- **Contribution and reflection:** member contribution/task data is stored in `contributions` → member submits periodic reflection linked by `contribution_id` → reflection status and submission timestamps track lifecycle.
- **Testimonial lifecycle:** member or admin creates a `testimonial_requests` row → testimonial content is generated/stored in `testimonials.content_json` and endorsement fields → finalization records `finalised_at`.
- **KPI reporting:** dashboard metrics are read through SQL functions such as `get_admin_kpi_active_events`, `get_admin_kpi_completion_rate`, `get_admin_kpi_tasks_due`, `get_admin_kpi_total_members`, `get_member_kpi_completion_rate`, `get_member_kpi_next_deadline`, `get_member_kpi_remaining_tasks`, and `get_member_kpi_team_sync_count`.
